    Introduction

    This manual describes how to add graphics, sound or animations to the different machines (also called 'targets') supported by DAAD Ready (ZX Spectrum, Commodore 64, PC/DOS, etc.)

    The first part of the manual explains some basic principles to understand why things differ depending on which machine/target we are using. The second explains how to include graphics, sounds or animations the way DAAD Ready expects them, and the third allows, in some machines, to go beyond.

    Original DAAD multimedia features

    DAAD had certain functionalities for sound and image playback:

    • It could display graphics, which on smaller machines (8-bit except Amstrad PCW) were vector-based (drawn progressively), and on larger machines (16-bit and PCW), digitized (more detailed images displayed instantly).
    • Those graphics could appear in windows—small or larger areas of the screen where the image was shown. Think of *Chichen Itza* on PC/Amiga/ST, which had a frame (covering the full screen), a box for each location graphic (a small window), and small boxes for inventory object images (several small windows).
    • DAAD allowed a graphic to be fixed (always shown in a predefined screen position) or floating (displayed inside the currently selected “window”).
    • For RGB-palette machines, DAAD also allowed choosing which colors from the graphic were loaded, making it possible to reserve some colors, for example, for text (so text color wouldn’t change depending on the image).
    • DAAD could also make sound, originally only through the BEEP command, which produced a synchronous beep (so background music wasn’t possible). Furthermore, BEEP wasn’t supported on all machines, so it wasn’t always usable.
    • On the other hand, the Atari ST and Amiga interpreters had limited but workable support for playing sound effects (small 8-bit mono samples).

    DAAD couldn’t play animations or video, although some Aventuras AD games showed simple ones by drawing several images successively in a window.

    Maluva Extension

    DAAD allows adding functionalities through an extension system called “EXTERN.” Maluva was one such extension, released years ago, that replaced—on 8-bit machines—the vector graphics typical of the time with more attractive digitized graphics, though it required disk or hard drive media.

    Maluva was also used to add support for certain ZX Spectrum variants: ZX Spectrum Next, ZX-Uno, and ZX Spectrum +DivIDE/MMC, enabling digitized graphics on those systems too.

    Maluva is now obsolete since its image rendering functionality has been integrated into the interpreters, making it unnecessary as an add-on.

    Still, when we speak today of “Maluva-style graphics,” we mean graphics displayed fixed at the top of the screen, spanning the full width—just like the original Maluva ones.

    The new interpreters

    After Maluva came the new interpreters—new software created in the 21st century, capable of running DAAD games on machines beyond the originals or on variations of them:

    • MSX 2 (MSX2DAAD): Enables use of MSX2 graphic modes.
    • PC/DOS (PCDAAD): Although there was an original DAAD interpreter for DOS, it only supported 16-color VGA mode, so a new one appeared supporting VGA and SVGA 256-color modes.
    • HTML (jDAAD): Generates a web page using HTML and JavaScript to run a DAAD adventure.
    • ZX Spectrum 128k (ZXDAAD128): Capable of running DAAD games with digitized graphics from cassette tape.
    • Windows and MacOS (ADP): Support the same graphics as Amiga or Atari ST. ADP is also an alternative interpreter for Amiga and Atari ST.

    There is also another interpreter capable of running—on any OS that supports Python—a game for any of the original DAAD platforms: NAPS, although it isn’t included in DAAD Ready.

    The new original interpreters

    New originals? Yes — thanks to the partial availability of the DAAD source code, some of the original interpreters have been recompiled, resulting in new ones based on the original sources but with added multimedia capabilities:

    • Commodore Plus/4: the new interpreter can load digitized graphics from disk, in the Maluva style, fixed at the top.
    • Spectrum Next: based on the original Spectrum sources, a new interpreter specifically for Spectrum Next allows fixed and floating 256-color graphics and limited palette loading (like Amiga and ST).
    • ZX-Uno: same functionality as Spectrum Next, using the Timex HiRes and ULAPlus modes to achieve high-quality images.
    • Spectrum +3: supports both fixed and floating digitized graphics.
    • Spectrum ESXDOS: supports both fixed and floating digitized graphics.
    • MSX1: supports loading fixed Maluva-style graphics.
    • CPC: supports loading fixed Maluva-style graphics, plus split-screen (see below).
    • Commodore 64: supports loading fixed Maluva-style graphics, plus split-screen (see below).
    • Commodore Plus/4: supports loading fixed Maluva-style graphics.

    Current status

    In short, depending on the interpreter used, DAAD Ready may or may not support the following features:

    • Floating graphics: with window support, floating and fixed graphics.
    • Digitized graphics, Maluva-style (only at the top of the screen)
    • Limited RGB-palette application
    • Sound via BEEP
    • Digitized sound effects
    • Synthesized or digitized background music
    • Animations
    • Extra messages (Xmessages)

    DAAD Ready and Multimedia

    DAAD Ready is a set of utilities and scripts for Windows that allow authors to focus on creating adventures, not on how to pack that adventure later into a disk or tape. DAAD Ready takes care of that.

    DAAD Ready also tries to simplify some things, because DAAD has many options, but in the end, the same ones are usually used. An advanced author may want to go into more detail, but DAAD Ready is not designed for them—although advanced authors can also use it.

    One of the simplifications we refer to is that although in DAAD graphics can be placed in many ways, DAAD Ready (by default, as we’ll see later) always follows the rule of putting the graphic at the top (in the first 96 scanlines of the screen) and the text below. This is because, on the one hand, it’s the most common layout in adventures, and on the other, because some interpreters can only display the image at the top of the screen and don’t support anything else (those that only have fixed graphics in the previous table).

    It is, of course, possible to place graphics in other parts of the screen, but maybe not digitized ones, or not on all machines/targets. Displaying a graphic 96 pixels tall and the full screen width is always possible on all platforms, so DAAD Ready sticks to that “maximum common denominator.”

    DAAD Ready also handles whether the graphics are fixed or floating, limited palette usage, etc., so the user doesn’t have to think too much about it.

    In the following sections we’ll explain how to add graphics and sound to DAAD Ready as set up by default. In another advanced section we’ll explain how to go beyond the limitations DAAD Ready imposes on itself.

    Multimedia Condacts

    There are several condacts and metacondacts closely related to DAAD Ready’s multimedia. These are:

    • PICTURE
    • DISPLAY
    • BEEP
    • XPLAY
    • SFX
    • GFX

    PICTURE is DAAD’s original condact to load graphics into memory (not to actually draw them). The one that really paints them on screen is DISPLAY. That is, first we select the graphic with PICTURE, then display it on screen with DISPLAY.

    BEEP is a simple sound condact. It takes duration and frequency as parameters, allowing for small sound effects or even melodies.

    XPLAY is a DAAD Ready compiler metacondact. It allows defining a melody in a slightly more human-readable way than with multiple BEEPs. For example, we can play Do, Re, Mi using English notation:

    XPLAY "CDE"

    In reality, no DAAD interpreter directly supports XPLAY; that’s why it’s called a metacondact. The DAAD compiler, when it encounters XPLAY, converts it into several BEEP commands, since BEEP is the only sound DAAD actually supports. So the interpreters receive three BEEP commands instead of one XPLAY, saving us from having to calculate frequencies manually.

    XPLAY uses a subset of a melody definition system called MML (Music Macro Language).

    SFX is a condact that integrates advanced sound functionality, originally supported only on Atari ST and Amiga. With DAAD Ready, PC/DOS and HTML now also support SFX—more advanced than the originals.

    GFX is a condact that integrates advanced graphical functions, originally supported only by the PC/DOS, Amiga, and ST interpreters. It’s now also supported on MSX2, the new PC/DOS interpreter, and HTML.

    Multimedia by platform

    Multimedia resources in DAAD Ready are identified by a number from 0 to 255, which is then used in DAAD code. For example, the PICTURE condact loads full-screen graphics, taking the resource number as a parameter. If we want to show the graphic for location 1, we use PICTURE 1 followed by DISPLAY 0.

    For each platform, you must place resources in the images or sounds folder and name them with three digits and the appropriate extension. For example, for Spectrum +3, place file 001.SCR in the IMAGES folder to show the graphic for location 1. For PC/DOS, place file 020.DRO in the SOUNDS folder, which can then be played as music resource 20.

    In the case of graphics, the DAAD Ready template automatically displays the corresponding image for each location—e.g., 001.SCR appears automatically when entering location 1.

    You can also create graphics not tied to locations, such as 200.SCR, which you can display manually from DAAD (using PICTURE 200).

    ZX Spectrum 48K

    This platform is the easiest to explain because DAAD Ready doesn’t support graphics for Spectrum 48K. The 48K Spectrum is too limited in RAM to load digitized graphics—only a few could fit—so the only way to add images is to use the old vector-drawing method, which falls outside DAAD Ready’s scope. For that, you should read the original DAAD manual.

    As for sound, DAAD supports BEEP on 48K, so sound can be done with BEEP or XPLAY.

    This machine doesn’t support animations or other sound types.

    Spectrum 128K (tape)

    The Spectrum 128K is an exceptional case within DAAD Ready, as it’s the only supported platform that supports graphics and generates cassette tape files (.TAP format).

    To add graphics, simply create .SCR files (DAAD Ready uses the 6912-byte .SCR format) and put them in the IMAGES folder. The SCR format is common on ZX Spectrum and consists of 6912-byte image files. To make the image appear automatically for a location, the file name must match the location number with three digits and the .SCR extension. You can also create numbered graphics not tied to locations to show them with PICTURE/DISPLAY.

    As it’s a tape-based system, all graphics must be preloaded into RAM—nobody would want to load from tape constantly. In practice, this means you’ll only fit about 15–25 graphics.

    The Spectrum 128K interpreter uses ZX0 compression to save space, which compresses better if:

    • The image has large areas of solid pixels or empty space when read horizontally.
    • Patterns repeat—for example, columns in a Greek temple repeating across scanlines compress better than unique lines.

    Because of the compression method, dithering tools like DaDither aren’t ideal for this system (though usable). Hand-drawn graphics made with ZX Paintbrush or Multipaint work best, though you can use dithering and correct manually.

    Sound-wise, Spectrum 128K supports BEEP and XPLAY. The sound plays through the 128K’s sound chip rather than the speaker.

    This machine doesn’t support animations or other sound types.

    ZX Spectrum +3

    With the Spectrum +3, memory space isn’t an issue since it loads graphics from disk on demand.

    The same graphics used for Spectrum 128K will load automatically on Spectrum +3.

    To add graphics, create .SCR files (6912 bytes) and place them in the IMAGES folder. To make the image appear automatically for a location, name the file with that location number (three digits) and the .SCR extension. You can also create manually displayed ones via PICTURE/DISPLAY.

    Compression isn’t used here, but disk space can be a problem. Each image takes about 3.3 KB, so only a limited number will fit on a 3" Spectrum +3 disk. Some authors used both sides of the disk and asked users to flip it when changing areas.

    For sound, Spectrum +3 supports BEEP and XPLAY via the 128K chip.

    This machine doesn’t support animations or other sound types.

    ZX Spectrum Next

    The Spectrum Next uses its Layer2 mode, 256x192 pixels, with 256 colors from a 512-color palette.

    To add graphics, create them in PCX format and put them in the IMAGES folder (e.g., 001.PCX for location 1).

    However, there’s a small problem: the Spectrum Next palette offers only 256 colors from 512, while PCX allows 256 colors from 262,144. This means you could use PCX colors not reproducible on Next.

    To avoid this, either:

    • When drawing manually, use this palette:
    • If colors are uncontrolled, use DaDither with “Spectrum Next / RGB333” palette mode and 256x192x8 settings to reduce properly, then export as PNG.

    Once you have a 256x192 image using only Spectrum Next palette colors, convert it to an indexed 256-color PCX. The easiest way is with Gimp: switch to indexed mode with a maximum of 240 colors (the remaining 16 are reserved for text colors INK 0–15). If it’s already indexed, toggle to RGB and back to ensure 240 max colors.

    NOTES:

    • Reducing from 256 to 240 may slightly degrade quality; try to limit manual artwork to 240 colors.
    • If Gimp detects fewer than 17 colors, it may create a 16-color PCX (invalid). Add dummy pixels below row 96 with extra colors to reach 17+ colors so Gimp saves a 256-color PCX.
    • Pure magenta (255,0,255) is treated as transparent on Next. Use it for transparent areas only.

    Once indexed, export as PCX (File → Export As) and name it with a 3-digit location number (e.g. 023.PCX).

    Sound support is the same as Spectrum +3/128K.

    This machine doesn’t support animations or other sound types.

    ZX Spectrum with DivIDE/DivMMC (ESXDOS)

    A DivIDE or DivMMC is an interface for ZX Spectrum that allows access to an SD or CF card as if it were a disk, although it is extremely faster than, for example, the +3 floppy disk. That’s why it is an excellent solution for loading digitized graphics in adventures.

    The graphics for ESXDOS (the DivMMC/DivIDE operating system) are the same as those used for Spectrum 128K (tape) or +3, so no further explanation is necessary.

    As for sound, DAAD supports BEEP on Spectrum with DivIDE(MMC), so you can make sound with BEEP or XPLAY. In this case, the sound will come from the speaker, since this interface can be connected to all types of Spectrums, including the 48K Spectrum.

    Finally, this machine does not support animations or other types of sound.

    ZX-Uno

    The ZX-Uno is a Spanish Spectrum clone that includes some features that appeared in other clones. For example, it supports a video mode called Timex HiColor, in which instead of having 8x8 pixel attributes where only one ink color and one paper color can exist, it uses 8x1 pixel attributes, allowing much more colorful graphics. In addition, it supports ULAPlus, a system that allows changing the default Spectrum colors and, in practice, using up to 64 colors on screen.

    Without reaching the Spectrum Next’s Layer2 mode, the combination of HiRes mode and ULAPlus allows for very high-quality graphics.

    To create graphics for ZX-Uno, we can do it in two ways:

    • We can draw them manually using, for example, ZX Paintbrush and Img2ULAPlus.
    • We can also use digitized images and process them again with DaDither, selecting platform ZX Spectrum ULAPlus, Screen 256x192x8 (8x1), and palette (Color, 48 high colors, slow). This will generate an image we can save from the Result menu in SCR format, which can be used directly in DAAD Ready. The “48 high colors” palette mode keeps the first 16 colors free, which, as in other systems, are reserved for text colors. Unlike the standard Spectrum, the generated files must be 12,288 bytes long instead of 6,912, and they must be saved in the IMAGES/ZXUNO folder to avoid overwriting files of the same extension but 6,912 bytes long, used by standard Spectrum interpreters.

    As for sound, DAAD supports BEEP on ZX-Uno, so sound can be made with BEEP or XPLAY. The sound will come from the Spectrum 128K sound chip.

    Finally, this machine does not support animations or other types of sound.

    MSX1

    To add images to the MSX1 interpreter, we need to place files with extension .SC2 (256x192) in the images folder.

    SC2 is a very common format on MSX, corresponding to a video mode similar to that of ZX-Uno but without palette switching.

    There’s a very good converter at https://msx.jannone.org/conv/. Make sure to select Screen 2 and the standard MSX1 palette.

    DaDither also allows creating images for MSX1 (Screen 2 256x192x4), and Multipaint includes an “MSX1 mode 2”.

    As for sound, DAAD supports BEEP on MSX1, so sound can be made with BEEP or XPLAY. The sound will come from the MSX sound chip.

    Finally, this machine does not support animations or other types of sound.

    MSX2

    MSX2 uses images in the standard MSX SC8 format (Screen 8), which allows an advanced 256-color palette and can be used at 256x212 resolution.

    We can also use the converter at https://msx.jannone.org/conv/, but selecting Screen 8 instead of Screen 2 as in MSX1.

    Another option is to use DaDither selecting platform MSX, Screen 8 256x212x8.

    DaDither also allows creating images for MSX2 (Screen 8 256x212x8).

    In Screen 8 mode, the MSX2 interpreter prevents the text from being affected by the graphic colors, so we don’t have to worry about that.

    Commodore 64

    To add images to the Commodore 64 interpreter, we need to place files with extension .ART (320x200) in the images folder. The ART format is very common on Commodore 64 for HiRes mode.

    Multipaint is an excellent utility for designing images directly in ART format.

    DaDither can also be used, but it’s not recommended, for two reasons:

    • It doesn’t achieve great quality with the limited color palette of the Commodore 64. Although better than ZX Spectrum, it’s still somewhat poor.
    • Commodore 64 disks are especially slow. Therefore, its graphics are compressed to reduce loading time. DaDither’s dithering compresses very poorly, which will negatively affect game performance.
    Split-screen mode

    DAAD originally works in the C64’s HiRes mode (320x200). Now it’s possible to use the Multicolor mode (160x200, but with more colors) at the top of the screen by enabling what is called “split screen.” This is done by editing the CONFIG.BAT file and changing where it says splitModeOff to splitModeOn.

    Unfortunately, maintaining that split mode running requires complex interrupt routines that are difficult to handle while reading from disk, so you’ll notice the entire screen is hidden while loading a new image, and you might even see some flickering or strange effects occasionally during disk reads or writes (for example, when saving or loading a game).

    In split-screen mode, instead of placing ART files in the IMAGES folder, you must place KOA files (Koala format) for Multicolor mode.

    You can use Multipaint or DaDither to generate KOA files.

    As for sound, DAAD supports BEEP, so sound can be made with BEEP or XPLAY. The sound will come from the SID sound chip.

    Finally, this machine does not support animations or other types of sound.

    Commodore Plus/4

    To add graphics, you must create Plus/4 PRG format images, which can be made using the Multipaint application.

    DaDither doesn’t support Plus/4, but you can use Pixel Polizei instead for similar functionality.

    As for sound, DAAD supports BEEP on Plus/4, so sound can be made with BEEP or XPLAY. The sound will come from the TED sound chip.

    Finally, this machine does not support animations or other types of sound.

    Amstrad CPC

    To add images to the Amstrad CPC interpreter, we need to place files with extension .SCR (320x200) in the images/CPC folder, together with a .PAL file of the same name (for example 001.SCR and 001.PAL). Do not place them directly in the images folder, as that would overwrite Spectrum SCR files, which have a different format.

    The SCR images must be in mode 1, 320x200, and 4 colors.

    Split screen

    As with Commodore 64, CPC can also enable split-screen mode, in which the upper part of the screen will be in mode 0 (160x200, 16 colors).

    To activate split-screen mode, you must edit CONFIG.BAT and change splitModeOff to splitModeOn.

    It will also experience flickering and hidden screen when loading from disk, so the image will turn off when loading a new one.

    The SCR/PAL format is the same regardless of whether you use mode 0 or 1. Simply generate an SCR for the mode you’ll use. You can use Multipaint or DaDither.

    Images should be mode 1 by default, but if you switch to split-screen, they must be 160x200 SCR files in mode 0.

    Another useful tool for CPC is ConvImgCPC, not only because it allows creating graphics like DaDither, but also because it allows editing the color palette. On Amstrad CPC, the palette selected for the graphic affects the text, so it’s important that our graphic has appropriate tones in the right color indexes. For example, color 0 should be black and color 7 white, or at least contrasting colors. This can be done in ConvImgCPC by locking those colors and forcing the tool to recalculate the image.

    As for sound, DAAD supports BEEP on CPC, so sound can be made with BEEP or XPLAY. The sound will come from the sound chip.

    Finally, this machine does not support animations or other types of sound.

    Commodore Amiga

    To add images to the Amiga interpreter, we’ll use the .PNG format, 320x200 with maximum 16 colo This is the standard Amiga image format and is also supported by Atari ST interpreters. You can create them with Multipaint or any graphics editor that can export IFF files (for example, GIMP).

    Recommended format: 320x256, 32 colors.

    Images can be placed in the IMAGES folder directly. Make sure the file names correspond to their location numbers (three digits plus the IFF extension).

    As for sound, the Amiga interpreters can play 8-bit mono samples via the SFX condact. This lets you add small effects like door sounds, footsteps, or similar cues.

    The Amiga does not support BEEP, but it does support SFX and GFX for special effects. Music must be imported as sound effects or through GFX if applicable.

    This machine doesn’t support animations beyond successive image displays.

    Atari ST

    The Atari ST uses the same image format as the Amiga: .IFF.

    Place them in the IMAGES folder and name them with three digits corresponding to the location number. The recommended resolution is 320x200 with 16 colors (the Atari ST’s typical low-resolution mode).

    Sound on Atari ST supports the SFX condact for playing 8-bit mono samples through the sound chip. BEEP is not supported, but SFX and GFX are available.

    This machine also does not support animations beyond image sequences.

    PC/DOS

    The PC/DOS interpreter is one of the most complete ones.

    It supports both VGA 256-color and EGA 16-color modes. Images should be in .BMP format at 320x200 (VGA) or 640x350 (EGA). The recommended format for DAAD Ready is VGA 320x200, 256 colors.

    Simply place the .BMP files in the IMAGES folder with three-digit numbers matching the location IDs. DAAD Ready will automatically use VGA mode.

    Sound on PC/DOS is handled through the .DRO format for AdLib (OPL2) music and .VOC files for digitized sound effects.

    • Background music: nnn.DRO files in the SOUNDS folder.
    • Sound effects: nnn.VOC files in the SOUNDS folder.

    You can play them via the SFX condact, referencing their resource number. For example, SFX 20 would play 020.VOC or 020.DRO, depending on which one exists.

    The PC/DOS interpreter also supports basic animations, by showing a sequence of BMPs from DAAD code.

    HTML (jDAAD)

    The HTML target is a very special one—it generates an HTML page and JavaScript that simulate the DAAD interpreter in a web browser. This allows publishing your adventure directly online.

    All images must be in .PNG format, ideally at 320x200 or similar size. Place them in the IMAGES folder using three-digit filenames, as usual.

    Sound files should be in .OGG or .MP3 format and placed in the SOUNDS folder. They are automatically loaded and can be played using SFX or BEEP/XPLAY if desired.

    Animations are supported by cycling through numbered image sequences—for example, 100_1.PNG, 100_2.PNG, 100_3.PNG, etc. jDAAD can display them smoothly in modern browsers.

    Windows

    For Windows, DAAD Ready uses the PC/DOS version under DOSBox by default, so the same rules as PC/DOS apply. However, there’s also an experimental interpreter called ADP that runs natively on Windows and can display the same 256-color graphics used by Amiga or ST.

    Images can be in .IFF or .BMP format. Sound can be in .WAV, .OGG, or .VOC.

    ADP supports the SFX and GFX condacts, allowing both sound effects and graphical transitions.

    MacOS

    The MacOS version also uses the ADP interpreter, just like Windows. The same rules apply for images and sound: - Images: .IFF or .BMP - Sounds: .WAV or .OGG

    It supports SFX and GFX condacts and behaves the same as on Amiga or ST, using the native Mac audio system.

    Loading screens

    Some interpreters allow displaying a loading screen while the game is initializing (before the first location is shown). DAAD Ready supports this feature automatically for interpreters that support it.

    To add a loading screen, simply place the image named 000.xxx (where xxx is the image format used on that platform, e.g. 000.SCR, 000.PCX, 000.BMP) in the IMAGES folder.

    That image will appear as soon as the game starts, before the first location is displayed.

    For interpreters that do not support loading screens, this image will be ignored.

    Advanced Multimedia by Platform

    Let’s now explain some features that the interpreters can handle, but which DAAD Ready—mainly to keep things simple—does not use by default.

    ZX Spectrum 128K and +3

    Support for graphics on Spectrum 128K and +3 is complete, meaning you can use fixed graphics (always appear in the same place, ignoring windows) and floating graphics (appear in the current window and are cropped to its size).

    If we look at ZXPLUS3.BAT or ZX128K.BAT, there are some lines that generate the graphics:

    for /L %%i in (0, 1, 9) do IF exist 00%%i.SCR ..\TOOLS\SCRMAKER\SCRMAKER spectrum 00%%i.SCR 00%%i.ZX 256 %IMGLINES% 0,0
    for /L %%i in (10, 1, 99) do IF exist 0%%i.SCR ..\TOOLS\SCRMAKER\SCRMAKER spectrum 0%%i.SCR 0%%i.ZX 256 %IMGLINES% 0,0
    for /L %%i in (100, 1, 256) do IF exist %%i.SCR ..\TOOLS\SCRMAKER\SCRMAKER spectrum %%i.SCR %%i.ZX 256 %IMGLINES% 0,0

    There’s one line for graphics 0–9, another for 10–99, and the last one for 100–255.

    What this line tells us, with that call to SCRMAKER, is that we want SCRMAKER to work in “spectrum” (or “spectrum128k”) mode, on the graphic xxx.SCR, generating an image 256 pixels wide and %IMGLINES% tall, fixed at position 0,0.

    First, IMGLINES is a value defined in CONFIG.BAT, whose default is 96. Try changing it in CONFIG.BAT to 48, for example, and you’ll see the graphics become half as tall. You can also set more, say 104, although in that case the text may overlap the graphic (since the DAAD template displays text at a fixed height, a taller graphic will overwrite it).

    Meanwhile, that 0,0 makes the 256x96 graphic fixed at position x=0, y=0 — the top-left corner. Being fixed means it ignores windows, always appearing at 0,0 (top-left of the screen) and 256 pixels wide — essentially a Maluva-style graphic.

    What if I want floating graphics, say for a graphical inventory? Let’s assume we’ll make all graphics above number 100 floating. Simply change the last line like this:

    for /L %%i in (100, 1, 256) do IF exist %%i.SCR ..\TOOLS\SCRMAKER\SCRMAKER spectrum %%i.SCR %%i.ZX 256 %IMGLINES%

    Note: that’s the line for Spectrum +3. In ZX128K.BAT it’s identical except it says “spectrum128k” instead of “spectrum”.

    This makes the graphics floating, since no coordinates are specified.

    “But I don’t want inventory graphics 256x96!”

    Then let’s change their width and height:

    for /L %%i in (100, 1, 256) do IF exist %%i.SCR ..\TOOLS\SCRMAKER\SCRMAKER spectrum %%i.SCR %%i.ZX 32 32

    This way, graphics 100–255 (if any) will be floating, 32x32 pixels in size.

    And what if I want a footer graphic—8 pixels tall, 256 pixels wide, always shown at the bottom?

    Let’s add a fourth line at the end, for graphic 200 for example:

    for /L %%i in (100, 1, 256) do IF exist %%i.SCR ..\TOOLS\SCRMAKER\SCRMAKER spectrum %%i.SCR %%i.ZX 256 8 0,184

    As you can see, it creates a 256x8 graphic at position x=0, y=184.

    Finally, there’s a feature supported by 16-bit machines and also by the Spectrum +3 interpreter. The original DAAD called them “buffered images”, which actually means preloaded images.

    A preloaded image is loaded at the beginning of the game and remains in memory until the end. This allows showing it instantly, very useful for frequently displayed or time-sensitive images—such as a graphical inventory or multiple images forming an animation.

    To make an image preloaded on +3, just put “L:” in the line that generates it, right before the ZX filename. For example, if we want image 200 preloaded, it would look like this (the first line for just 200, the second for any from 100–255):

    IF exist 200.SCR ..\TOOLS\SCRMAKER\SCRMAKER spectrum 200.SCR L:200.ZX 256 8 0,184
    for /L %%i in (100, 1, 256) do IF exist %%i.SCR ..\TOOLS\SCRMAKER\SCRMAKER spectrum %%i.SCR L:%%i.ZX 256 8 0,184

    You might wonder why we don’t make all images preloaded. There are two reasons:

    • Memory is limited on Spectrum +3, so loading everything is impossible in many cases.
    • If all images were preloaded, the game would take much longer to start, since it’d load them all initially.

    That’s why the preloaded image system has only 8,960 bytes available. You must choose carefully what to preload, as exceeding this will unnecessarily lengthen the game’s startup.

    For obvious reasons, in the 128K tape version, all images are preloaded.

    ZX Spectrum with DivMMC/DivIDE Interface

    Support on a ZX Spectrum with DivIDE/MMC is similar to Spectrum +3 and 128K, so refer to the previous section. Spectrum Next doesn’t support preloaded images, since they aren’t needed—loading from the SD card is extremely fast.

    ZX Spectrum Next

    Support on Spectrum Next is very similar to Spectrum +3, though with some differences:

    Let’s look at this example:

    for /L %%i in (0, 1, 9) do IF exist 00%%i.PCX ..\TOOLS\SCRMAKER\SCRMAKER specnext 00%%i.PCX 00%%i.LY2 256 %IMGLINES% 0,0 /s
    for /L %%i in (10, 1, 99) do IF exist 0%%i.PCX ..\TOOLS\SCRMAKER\SCRMAKER specnext 0%%i.PCX 0%%i.LY2 256 %IMGLINES% 0,0 /s
    for /L %%i in (100, 1, 256) do IF exist %%i.PCX ..\TOOLS\SCRMAKER\SCRMAKER specnext %%i.PCX %%i.LY2 256 %IMGLINES% 0,0 /s

    The Spectrum Next uses a color palette. That means color 0 isn’t necessarily black—it depends on how color 0 is defined in the palette. That’s why, apart from replacing “spectrum” with “specnext”, we see the /s parameter. What does it do?

    It tells the image converter to shift the palette up by 16 colors and replace the first 16 colors with the standard Spectrum colors. In practice, this prevents the graphic from altering text colors, which normally use those first 16 palette slots.

    Remember when we said to use only 240 colors in the PCX file? That’s why. With colors 0–239 used, shifting them 16 steps up moves them into the 16–255 range, leaving room for the 16 standard text colors. If the image had more than 240 colors, there’d be no room left for those text colors.

    Is it mandatory? No—if you remove /s, the shift won’t happen, and all 256 colors can be used, but then you must ensure color 0 is black, 7 is white, etc., so text displays correctly.

    Of course, we can still make fixed graphics in other positions, floating ones, or of different sizes, the same way as on Spectrum +3 and ESXDOS.

    Another feature not visible in Spectrum +3, 128K, or ESXDOS targets is palette limiting. This is done by adding a parameter indicating the first and last color numbers separated by a dash. For example, this line limits the palette to colors 0–127:

    for /L %%i in (100, 1, 256) do IF exist %%i.PCX ..\TOOLS\SCRMAKER\SCRMAKER specnext %%i.PCX %%i.LY2 256 %IMGLINES% 0,0 0-127

    This allows displaying two images without their colors interfering—each one uses a different palette range. This is essential for graphical inventories. If you draw a frame, a location image, and six object graphics simultaneously, all images must share a palette or each occupy different ranges; otherwise, their colors will overwrite each other, and the display will look wrong.

    It’s even possible to make an image ignore palette changes entirely by using an inverted range (second number lower than the first):

    for /L %%i in (100, 1, 256) do IF exist %%i.PCX ..\TOOLS\SCRMAKER\SCRMAKER specnext %%i.PCX %%i.LY2 256 %IMGLINES% 0,0 1-0

    The Layers of the Spectrum Next

    Unlike the Spectrum +3—or any other machine—the Spectrum Next uses a system of layers on screen, of which DAAD only uses two. A layer system means there are several “virtual” screens that are composited to the final display. Each layer can have transparent regions, so the layers beneath remain visible.

    DAAD uses two layers on the Spectrum Next:

    • The ULA layer (the original Spectrum screen), used for text and drawn first.
    • The Layer2 layer, allowing 256-color graphics, drawn on top of the ULA layer.

    When DAAD starts, Layer2 is fully transparent, so only the ULA layer is visible. When graphics are drawn, they’re placed on Layer2. DAAD Ready automatically ensures transparency outside the graphic area so the text below remains visible. However, if we want to place graphics elsewhere (for example, to build a graphical inventory) or add a decorative frame, we must ensure manually that transparent areas are properly defined. To do that, simply use pure magenta (RGB 255,0,255 or #FF00FF) for any area you want to be transparent. That color will make the underlying ULA text visible.

    Of course, that pure magenta color shouldn’t appear in actual visible graphics, since it would become transparent.

    ZX-Uno

    The ZX-Uno is also a machine with a palette, so its functionality is very similar to the Spectrum Next’s, although it lacks the layer system:

    for /L %%i in (0, 1, 9) do IF exist 00%%i.SCR ..\..\TOOLS\SCRMAKER\SCRMAKER zxuno 00%%i.SCR 00%%i.ZXU 256 %IMGLINES% 0,0 16-63 /SCR
    for /L %%i in (10, 1, 99) do IF exist 0%%i.SCR ..\..\TOOLS\SCRMAKER\SCRMAKER zxuno 0%%i.SCR 0%%i.ZXU 256 %IMGLINES% 0,0 16-63 /SCR
    for /L %%i in (100, 1, 256) do IF exist %%i.SCR ..\..\TOOLS\SCRMAKER\SCRMAKER zxuno %%i.SCR %%i.ZXU 256 %IMGLINES% 0,0 16-63 /SCR

    The main difference is that “zxuno” is used instead of “spectrum”, and the /SCR parameter tells SCRMAKER that the input graphic is SCR format (as produced by DaDither). If omitted, it expects MLT (as produced by Img2ULAPlus). The 16–63 color range matches what we described earlier for ZX-Uno graphics in DaDither—reserving the first 16 colors for text, the rest for graphics. There’s no palette shift here; the image must already avoid using the first 16 colors. The /s parameter does not work on ZX-Uno.

    We can still limit palette use to any range we want, for example to make a graphical inventory.

    MSX2

    MSX2 uses the new MSX2DAAD interpreter, which has some specific behaviors regarding graphics:

    • Regarding traditional DAAD graphics, it only supports one type—a hybrid between floating and fixed:
      • Like fixed graphics, it applies the palette and ignores window size.
      • Like floating ones, it appears positioned in the current window.
    • It supports transparency, allowing parts of the image to remain invisible (useful for overlays).
    • It doesn’t support partial palette application.

    Because of this, working with advanced graphics on MSX2 differs slightly:

    for /L %%i in (0, 1, 9) do IF exist 00%%i.SC8 IF NOT exist 00%%i.IM8 ..\php\php ..\tools\imgwizard\imgwizard.php c 00%%i.SC8 %IMGLINES% RLE
    for /L %%i in (10, 1, 99) do IF exist 0%%i.SC8 IF NOT exist 0%%i.IM8 ..\php\php ..\tools\imgwizard\imgwizard.php c 0%%i.SC8 %IMGLINES% RLE
    for /L %%i in (100, 1, 255) do IF exist %%i.SC8 IF NOT exist %%i.IM8 ..\php\php ..\tools\imgwizard\imgwizard.php c %%i.SC8 %IMGLINES% RLE

    We can add a transparency color—an index from 0 to 255. For example, to make color 28 transparent:

    for /L %%i in (100, 1, 255) do IF exist %%i.SC8 IF NOT exist %%i.IM8 ..\php\php ..\tools\imgwizard\imgwizard.php c %%i.SC8 %IMGLINES% RLE 28

    We can also select a rectangular region—for example, one at position 0,0 with width and height 64:

    for /L %%i in (100, 1, 255) do IF exist %%i.SC8 IF NOT exist %%i.IM8 ..\php\php ..\tools\imgwizard\imgwizard.php s %%i.SC8 0 0 64 64

    Note how we changed the “c” after imgwizard.php to an “s”.

    We can also select a rectangle at x=20, y=30 with width 128 and height 32, with transparency color 6:

    for /L %%i in (100, 1, 255) do IF exist %%i.SC8 IF NOT exist %%i.IM8 ..\php\php ..\tools\imgwizard\imgwizard.php s %%i.SC8 20 30 128 32 6

    MSX2 doesn’t support palette limiting. Although text is unaffected by the image palette, multiple graphics must share the same palette to appear together.

    Additionally, MSX2 supports the GFX condact—one of the few interpreters that do. GFX uses double buffering and is invoked with a function number as the second parameter and, optionally, extra data in the first. For example, GFX x 0 copies the buffer to the screen, regardless of x.

    Here’s the list of GFX functions:

    Parameter 2Parameter 1Description
    0AnyCopies the buffer to the screen
    1AnyCopies the screen to the buffer
    2AnySwaps the contents of the buffer and the screen
    3AnySets drawing to occur directly on the screen
    4AnySets drawing to occur on the buffer
    5AnyClears the screen
    6AnyClears the buffer
    9flagnoChanges the palette color defined in flag flagno using RGB values stored in flags flagno+1, flagno+2, and flagno+3.
    10flagnoReads the palette color defined in flag flagno and stores its RGB values in flags flagno+1, flagno+2, and flagno+3.
    128AnyCopies the current window from the buffer to the screen
    129AnyCopies the current window from the screen to the buffer

    Atari ST

    Atari ST has full graphics support, but the way to do things is different from the Spectrum +3 or Spectrum Next. Unlike those, if we want floating graphics or to limit the palette, we need to add a file next to the graphic — a text file with the extension .json and the same number. For example, if we want to change things for graphic 020.PNG, we must create a file named 020.json and place it in the same folder as the graphic. The content of that file would be something like this (three examples are given):

        {"X":24,"Y":180,"PCS":0,"PCE":15}
    
    {"float":1,"buffer":1, "width":120, "height":96}
    
    {"float":1}
    • In the first example, we indicate a fixed graphic located at 24,180, and the palette range to apply is 0–15 (which is the default anyway).
    • In the second example, we say that the graphic is floating, that it has dimensions of 120x96, and that it should be buffered (this means it loads into RAM when the game starts and is always available without reading the disk each time).
    • The third example simply states that the graphic is floating.

    The file options are as follows:

    • float:0–1 → 0 = fixed image, 1 = floating image
    • buffer:0–1 → 0 = not buffered, 1 = buffered (loads into RAM at game start)
    • X:0–255 → X position of the graphic on screen (only if fixed)
    • Y:0–255 → Y position of the graphic on screen (only if fixed)
    • PCS:0–15 → first color of the palette to apply (only if fixed)
    • PCE:0–15 → last color of the palette to apply (only if fixed)
    • width:0–255 → width of the graphic (only if floating)
    • height:0–255 → height of the graphic (only if floating)

    Finally, Atari ST supports the following GFX options:

    Parameter 2Parameter 1Description
    0AnyCopies the buffer to the screen
    1AnyCopies the screen to the buffer
    2AnySwaps the contents of buffer and screen
    3AnySets graphics drawing to the screen
    4AnySets graphics drawing to the buffer
    5AnyClears the screen
    6AnyClears the buffer
    7AnySets text drawing to the screen
    8AnySets text drawing to the buffer
    9flagnoChanges the palette color defined in flag flagno to the R,G,B values contained in flags flagno+1, flagno+2, and flagno+3.
    10flagnoReads the palette color defined in flag flagno and stores its R,G,B values into flags flagno+1, flagno+2, and flagno+3.

    PC/DOS

    In DOS, the graphics generation is done by these lines in MSDOS.BAT:

    for /L %%i in (0, 1, 9) do IF exist 00%%i.PCX %IMGPATHBACK%\TOOLS\SCRMAKER\SCRMAKER.exe msdos 00%%i.PCX 00%%i.MSD %IMGWIDTH% %IMGHEIGHT% 0,0 0-255 /s
    for /L %%i in (10, 1, 99) do IF exist 0%%i.PCX %IMGPATHBACK%\TOOLS\SCRMAKER\SCRMAKER.exe msdos 0%%i.PCX 0%%i.MSD %IMGWIDTH% %IMGHEIGHT% 0,0 0-255 /s
    for /L %%i in (100, 1, 256) do IF exist %%i.PCX %IMGPATHBACK%\TOOLS\SCRMAKER\SCRMAKER.exe msdos %%i.PCX %%i.MSD %IMGWIDTH% %IMGHEIGHT% 0,0 0-255 /s

    This code says to use the MSDOS model, for width %IMGWIDTH% and height %IMGHEIGHT%, creating fixed graphics at x=0, y=0, applying palette 0–255, and shifting the palette to leave room for text colors (the first 16 palette colors). The result is a 256x96 fixed graphic at position 0,0, with a shifted palette. That’s why we said earlier that graphics should have 240 colors—to leave room for the 16 text colors without affecting the graphic.

    As you can see, the model is very similar to Spectrum Next: fixed graphic positions can be set with two numbers separated by commas, and palette ranges with two numbers separated by dashes. As on the Next, if we don’t want a palette applied at all, we can set an invalid range like 1-0. Similarly, to make a floating graphic, just remove the X and Y coordinates.

    Some examples:

    for /L %%i in (100, 1, 256) do IF exist %%i.PCX %IMGPATHBACK%\TOOLS\SCRMAKER\SCRMAKER.exe msdos %%i.PCX %%i.MSD 320 96 128-255
    for /L %%i in (100, 1, 256) do IF exist %%i.PCX %IMGPATHBACK%\TOOLS\SCRMAKER\SCRMAKER.exe msdos %%i.PCX %%i.MSD 32 32 1-0
    for /L %%i in (100, 1, 256) do IF exist %%i.PCX %IMGPATHBACK%\TOOLS\SCRMAKER\SCRMAKER.exe msdos %%i.PCX %%i.MSD 320 8 0,192 0-255
    • The first marks a floating graphic, 320x96 in size, applying palette 128–255.
    • The second marks a floating graphic, 32x32 in size, with no palette applied.
    • The third marks a fixed graphic, 320x8, located at x=0, y=192, applying palette 0–255.

    In addition, the PC/DOS interpreter can play samples similar to those on Amiga and ST (using PICTURE and SFX 0 255).

    In the HTML BAT file, we can see the lines that generate images for jDAAD:

    for /L %%i in (0, 1, 9) do IF exist IMAGES\HTML\00%%i.png PHP\PHP TOOLS/jDAADImager/jDAADImager.php IMAGES\HTML\00%%i.png RELEASE\HTML\images.js 0,0

    HTML doesn’t support palette limits—but it doesn’t need to, since browsers can display multiple images each with its own palette without any problem.

    However, we can specify whether an image is fixed or floating, by including or omitting the pair of comma-separated numbers at the end. If they appear, they define a fixed position; if not, the image is floating:

    for /L %%i in (100, 1, 256) do IF exist IMAGES\HTML\%%i.png PHP\PHP TOOLS/jDAADImager/jDAADImager.php IMAGES\HTML\%%i.png RELEASE\HTML\images.js 50,50
    for /L %%i in (100, 1, 256) do IF exist IMAGES\HTML\%%i.png PHP\PHP TOOLS/jDAADImager/jDAADImager.php IMAGES\HTML\%%i.png RELEASE\HTML\images.js
